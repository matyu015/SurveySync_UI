# Database Design & Schema Concepts

SurveySync utilizes Firebase Firestore, a **NoSQL document database**. Unlike traditional SQL databases (like MySQL or PostgreSQL), NoSQL databases do not use tables, rows, and strict relational foreign keys. Instead, they use **Collections** and **Documents**.

## 1. Core NoSQL Concepts (How it relates to SQL)
- **Collection** = Table (e.g., `users`, `requests`)
- **Document** = Row (A single record inside a collection)
- **Field** = Column (Data inside a document, e.g., `firstName`, `email`)

## 2. Primary and Foreign Keys in Firestore
Because Firestore is NoSQL, it does not enforce strict referential integrity (foreign keys) at the database level. Instead, we manage relationships within the application code.

### Primary Keys
Every document in a Firestore collection is automatically assigned a unique ID (a long string of characters) upon creation.
- **Example:** `users/j9XkL2...` (where `j9XkL2...` is the Primary Key for that user).
- **In Auth:** Often, the Document ID in the `users` collection is explicitly set to match the UID (User ID) generated by Firebase Authentication.

### Foreign Keys
To link documents together (e.g., linking a request to a user), we store the Primary Key of one document as a string field in another document.
- **Example:** In the `requests` collection, the field `clientId` stores the Document ID of the user who made the request. This acts as a foreign key.

## 3. SurveySync Database Schema Classifications

The database is divided into 5 primary collections:

### Collection 1: `users`
**Purpose:** Stores user profiles and permissions.
- **Document ID:** User's Auth UID (Primary Key)
- `firstName` (String): User's first name.
- `lastName` (String): User's last name.
- `email` (String): User's email address.
- `role` (String): Classification of the user (e.g., `admin`, `client`). This is crucial for RBAC (Role-Based Access Control).
- `createdAt` (Timestamp/String): When the account was made.

### Collection 2: `requests`
**Purpose:** Tracks survey requests submitted by clients.
- **Document ID:** Auto-generated ID (Primary Key)
- `clientId` (String): Reference to the `users` collection (Foreign Key).
- `clientName` (String): Denormalized data to avoid extra lookups.
- `surveyType` (String): Classification of the survey needed.
- `status` (String): The state of the process (e.g., `submitted`, `in-progress`, `completed`).
- `referenceNo` (String): A human-readable tracking number.

### Collection 3: `payments`
**Purpose:** Tracks financial transactions related to requests.
- **Document ID:** Auto-generated ID (Primary Key)
- `clientId` (String): Reference to the `users` collection (Foreign Key).
- `amount` (Number): The payment amount.
- `status` (String): Payment state (e.g., `pending`, `paid`).
- `referenceNo` (String): Receipt or tracking number.

### Collection 4: `availability`
**Purpose:** Manages scheduling for administrators.
- **Document ID:** Auto-generated ID (Primary Key)
- `date` (String): Calendar date.
- `startTime` (String): Start of available slot.
- `endTime` (String): End of available slot.
- `status` (String): Slot state (e.g., `available`, `booked`).

### Collection 5: `documents`
**Purpose:** Maps uploaded files to specific requests.
- **Document ID:** Auto-generated ID (Primary Key)
- `name` (String): File name.
- `fileUrl` (String): The public URL to the file hosted in Supabase Storage.
- `requestId` (String): Reference to the `requests` collection (Foreign Key).
